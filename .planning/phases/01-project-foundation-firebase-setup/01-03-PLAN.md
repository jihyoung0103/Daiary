<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| app/google-services.json → Firebase SDK | JSON file contains Firebase project API key and project ID; processed by google-services plugin at build time |
| AndroidManifest.xml → Android OS | Manifest declares permissions, activities, and exported components; incorrect exported=true can expose components to other apps |
| nav_graph.xml → NavController | Navigation graph defines all possible destinations and actions; incorrect startDestination can bypass auth gate |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Information Disclosure | google-services.json | mitigate | File added to .gitignore in this plan; Firebase API keys are low-risk alone but combined with open Firestore rules would expose all user data; Security Rules hardened in Phase 4 |
| T-03-02 | Elevation of Privilege | android:exported="true" on MainActivity | accept | MainActivity is the launcher Activity and must be exported; it does not accept untrusted Intent extras in Phase 1; Phase 2+ must validate any Intent data before use |
| T-03-03 | Spoofing | nav_graph.xml startDestination | mitigate | loginFragment is the start destination; Phase 3 adds authStateChanges() check in MainActivity before routing to calendarFragment; no unauthenticated path to calendarFragment exists by default |
| T-03-04 | Information Disclosure | strings.xml error messages | accept | Error strings reveal authentication failure reasons (wrong password, user not found); this is standard UX — the alternative (generic "login failed") creates worse UX without meaningful security benefit for email/password auth |
</threat_model>
